# Tempo C++ Client API

Drop-in C++ client library for the Tempo simulation server. Generated by
`gen_cpp_api.py` at Tempo build time when `TEMPO_GEN_CPP_API=1`.

## What's in this folder

```
include/
  tempo.h                     # umbrella header — include this and you're done
  tempo/                      # the high-level client surface
    context.h                 # set_server, TempoContext
    error.h                   # TempoError
    result.h                  # Result<T>
    streaming.h               # ServerStream<T>
    <service>.h               # one per gRPC service (tempo_core, tempo_sensors, ...)
  proto/
    <Package>/<File>.pb.h     # protoc-generated message types
    <Package>/<File>.grpc.pb.h# protoc-generated service stubs (advanced use only)
lib/
  <Platform>/libtempo.{a,lib} # static archive — wrapper + protoc outputs
proto/
  <Package>/<File>.proto      # decorated source protos (informational; for users
                              # who want to regenerate against a different gRPC)
```

## Integration

1. Add to your include path:
   - `<this-folder>/include`
   - `<this-folder>/include/proto`
2. Link against `lib/<Platform>/libtempo.{a,lib}` (the archive for your build host).
3. Bring your own gRPC++ and protobuf — `libtempo` does **not** bundle them. You
   have three reasonable options:
   - **Vendored from Tempo** (recommended; guaranteed ABI match): use the
     headers and static archives at
     `Plugins/Tempo/TempoCore/Source/ThirdParty/gRPC/{Includes,Libraries/<Platform>}`.
   - **vcpkg / Conan**: install gRPC via your package manager. ABI compatibility
     depends on flags — see "ABI considerations" below.
   - **System gRPC** (`apt install libgrpc++-dev`, etc.): same caveat.
4. Define the same gRPC compile flags used to build this library:
   ```
   GOOGLE_PROTOBUF_NO_RTTI=1
   GRPC_ALLOW_EXCEPTIONS=0
   ```

## Quick start

```cpp
#include <tempo.h>

int main() {
    tempo::set_server("localhost", 10001);

    auto result = tempo::tempo_core::get_current_level_name();
    if (!result) {
        std::fprintf(stderr, "request failed: %s\n", result.error().what().c_str());
        return 1;
    }
    std::printf("current level: %s\n", result.value().level().c_str());
    return 0;
}
```

## Streaming RPCs

```cpp
auto stream_result = tempo::tempo_sensors::stream_color_images(
    /*owner=*/"Vehicle", /*sensor=*/"Cam0", /*compression_level=*/0);
if (!stream_result) { /* handle error */ }
auto stream = std::move(stream_result).value();

TempoSensors::ColorImage frame;
while (stream.read(&frame)) {
    // process frame
}
auto status = stream.finish();
```

## ABI considerations

`libtempo` was compiled against the gRPC version that ships in the Tempo
plugin. The protobuf-generated stubs inside the archive are tightly coupled to
that gRPC version's ABI. If you link against a different gRPC version, expect
runtime failures or build-time symbol mismatches.

To regenerate the library for a different gRPC, point `gen_cpp_api.py` at a
different `TEMPO_GRPC_DIR`, or use the `.proto` files in `proto/` as input to
your own codegen pipeline.

## Multi-platform

This folder ships one `lib/<Platform>/` slot per build host that's run
`TEMPO_GEN_CPP_API=1`. To populate all three platforms (Mac, Win64, Linux),
build on each platform and commit the resulting `libtempo.{a,lib}`.

## Build prerequisites for regeneration

If you're regenerating with `TEMPO_GEN_CPP_API=1`:
- `cmake` 3.20 or later on `PATH`.
- A C++20-capable host compiler (clang 14+, gcc 11+, or MSVC 19.30+).
- The Tempo Python venv (set up automatically by Tempo's prebuild).

Consumers who link against `libtempo` must also build their code with C++20:
the vendored gRPC's absl headers depend on `std::weak_ordering` and other
C++20 spaceship-operator types.
